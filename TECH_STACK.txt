AI FACE DETECTION
  - Technology: TensorFlow.js + MediaPipe Face Detection Model
  - Runtime: Every 1 second during exam
  - Detection: Student's face presence in camera feed
  - Violation: No face detected for 5+ seconds
  - Processing: Local (GPU accelerated, not sent to server)

FULLSCREEN ENFORCEMENT
  - Exam must be in fullscreen mode
  - Instant violation if student exits fullscreen
  - Monitored via document.fullscreenElement

TAB SWITCHING DETECTION
  - Watches document.visibilitychange event
  - Prevents looking at external resources/notes
  - Violation on visibility change

FOCUS LOSS DETECTION
  - Monitors window.blur event
  - Detects extensions and popups stealing focus
  - Violation on any focus loss

KEYBOARD SHORTCUT BLOCKING
  - Blocked: Ctrl+J (Console), Ctrl+U (Source), Ctrl+S (Save)
  - Blocked: Ctrl+P (Print), Ctrl+I (DevTools), Ctrl+F (Find)
  - Blocked: F12 (DevTools), Ctrl+G (Find Next)
  - Warning: Alt/Ctrl/Cmd key combinations displayed

DEVTOOLS PREVENTION
  - F12 key is blocked
  - Prevents console and inspector access
  - Stops element inspection

SIDE-PANEL EXTENSION DETECTION
  - Checks every 2 seconds: screen.width - innerWidth > 100px
  - Detects browser extensions with sidebars
  - Detects developer tools panels

EXTENSION/IFRAME OVERLAY DETECTION
  - Uses MutationObserver for new DOM nodes
  - Detects injected iframes
  - Blocks overlay and sidebar attempts

CLIPBOARD OPERATIONS BLOCKING
  - Disabled: Right-click, Copy, Cut, Paste
  - Blocked: Drag & Drop, Text selection via drag
  - Prevents information copying

SCREEN RESIZE DETECTION
  - Monitors window.resize events

VIOLATION TRACKING & PENALTIES

CODING MODULE - CODE EVALUATION SYSTEM
================================================================================

OVERVIEW
  - Module Type: Competitive Programming Style Exam Platform
  - Frontend: React with Monaco Editor (for code editing)
  - Backend: Node.js with child_process for code execution
  - Evaluation: Automated test case matching

SUPPORTED PROGRAMMING LANGUAGES
  1. Python
  2. C
  3. C++
  4. Java
  (Currently only Python fully implemented)

QUESTION STRUCTURE (CodingQuestion Model)
  - Title: Question name
  - Description: Problem statement
  - Difficulty: EASY, MEDIUM, HARD
  - Function Name: Expected function name to implement
  - Parameters: Array of input parameters with name and type
  - Return Type: Expected return type
  - Marks: Points awarded for correct solution
  - Sample Test Cases: Visible to student during exam (for testing)
  - Hidden Test Cases: Used for evaluation (max 5 test cases)

EVALUATION FLOW

Step 1: RUN (Sample Test Case)
  - Endpoint: POST /api/student/exams/:examId/coding/run
  - Input: Code + Language + Question ID
  - Process:
    a) Fetch first sample test case
    b) Execute code with sample input locally
    c) Return output to student
  - Purpose: Allows student to test code before submission
  - Uses: localRunner.js with Python subprocess execution

Step 2: SUBMIT QUESTION (Hidden Test Cases)
  - Endpoint: POST /api/student/exams/:examId/coding/submit
  - Input: Code + Language + Question ID
  - Evaluation Process:
    a) Fetch all hidden test cases (1-5 cases)
    b) Execute code against each test case sequentially
    c) Compare actual output with expected output
    d) Calculate marks: (passed/total) * questionMarks
  - Output: Marks awarded, test cases passed count
  - Storage: Answer saved in ExamAttempt with code and marks

Step 3: FINAL SUBMIT (Exam Completion)
  - Endpoint: POST /api/student/exams/:examId/submit-final
  - Process:
    a) Accumulate all question scores
    b) Mark exam as SUBMITTED or AUTO_SUBMITTED (if 3 violations)
    c) Record end time
  - Score Calculation: Sum of all individual question marks

CODE EXECUTION (localRunner.js)

Technical Implementation:
  - Method: Node.js spawn() - Executes code in separate process
  - Timeout: 5 seconds per execution
  - File Handling: Creates temporary .py file with UUID name
  - Input Handling: Passes input via stdin to child process
  - Output Capture: Captures stdout and stderr
  - Cleanup: Deletes temporary file after execution

Execution Steps:
  1. Generate unique filename: temp_${uuid}.py
  2. Write code to temporary file
  3. Spawn Python process with file path
  4. Send input (stdin)
  5. Capture output (stdout) and errors (stderr)
  6. Kill process on completion or timeout
  7. Parse and return output
  8. Delete temporary file
  9. Handle errors and edge cases

Output Handling:
  - Success: Returns { success: true, stdout: "...", stderr: "" }
  - Failure: Returns { success: false, stdout: "", stderr: "..." }
  - Trimmed: All output trimmed to avoid whitespace issues

PYTHON EXECUTION ARCHITECTURE

Where Code Execution Happens:
  Browser: NOT runs Python ❌
  Backend Server: YES runs Python ✅
  
Communication Flow:
  1. Browser (React) sends code via HTTP
  2. Backend (Node.js) receives code
  3. Backend creates temp Python file
  4. Backend uses spawn() to run Python
  5. Python executes the code
  6. Backend captures output
  7. Backend sends result back to Browser
  8. Browser displays marks

How We Ask Python to Run Code:
  - Tool: Node.js spawn() function from child_process
  - Location: /backend/utils/localRunner.js
  - Command: spawn("python", [filePath])
  - Equivalent to typing in terminal: $ python script.py

Key Technical Points:

Browser Limitations:
  ❌ Cannot run Python directly
  ❌ Cannot create/delete files
  ❌ Cannot execute system commands
  ❌ Cannot access file system
  ❌ Cannot install Python modules
  ✅ Only runs JavaScript

Server Capabilities:
  ✅ Can run any installed language
  ✅ Can create/delete files
  ✅ Can execute system commands
  ✅ Can access file system
  ✅ Can manage processes
  ✅ Can compare outputs

Why Server Handles Code Execution:
  - Security: Code runs in isolated process on server
  - Compatibility: Python already installed on server
  - Control: Server manages timeout and resource limits
  - Consistency: Same Python version for all students

TEST CASE COMPARISON

Matching Logic:
  - Compares: actual output vs expected output (string comparison)
  - Trimmed: Both sides trimmed to handle whitespace
  - Passed Count: Number of test cases with correct output
  - Partial Credit: Yes (e.g., 2/5 test cases = 40% marks)

Example:
  Question: 3 marks, 5 hidden test cases
  If student passes 3/5 test cases: Marks = (3/5) * 3 = 1.8 ≈ 1.8 marks
